破坏开发者生产力的12件事

 2021.12.2  2021.12.8  3578  8 分钟

很多文章都是针对技术负责人和工程经理的角色。我们经常遇到的一个共同主题是如何提高一个团队的生产力。但是，在你集中精力试图提高他们的生产力之前，你可能首先要考虑是什么在破坏它，以便你可以建立一个健全的基础。不幸的是，尽管Peopleware是在近30年前出版的，但我们看到很多团队以一些（负面的）显著的方式遭受巨大的生产力损失

没有人期望程序员在没有接触电脑的情况下完成工作，但有许多公司期望程序员在没有了解他们想法的情况下完成工作。这同样是不现实的。

因此，让我们深入探讨一下我们的清单，即阻碍你的开发人员 "进入状态 "和提高生产力的12件事情。我将尝试把这个列表从最有影响的到最没有影响的进行优先排序。请自由评论!

如果你想知道这一切是否值得投资，只要考虑一下开发人员的工资。即使是增加10%的生产力也是一个很大的数目

1) 中断与会议

在我看来，中断是开发人员的首要生产力杀手。开发人员不能轻易回到他们在中断之前的位置。他们需要进入开发的心态，然后慢慢追溯到他们离开的地方。这很容易需要30分钟以上的时间。而更多的中断导致更多的挫折，更少的工作质量，更多的bug--这一切都在循环往复。

"当我试图开始工作时，你给我的绊脚石越多--我每次尝试的间隔就越长。如果你让我的早晨充满了干扰--当这一天没有收获的时候，请不要感到惊讶。"Reddit上的一位开发者

那么会议呢？会议和中断的唯一区别是，会议是有计划的中断，这使得情况更加糟糕。如果开发人员知道他们在工作中会有中断，他们就无法在任务上取得进展。因此，如果他们在一两个小时后有一个会议，他们将无法在任何事情上取得进展，因为大多数工程任务需要更多的时间。

正如保罗-格雷厄姆所写的那样，"一次会议就可以把整个下午的时间分成两块，每块都太小，无法做任何困难的事情。"

这怎么能避免呢？这一部分是有据可查的；你没有任何借口。例如，在一天刚开始时或在午餐前举行简短的状态会议，以避免不必要的中断。

2) 微观管理

在不同类型的管理者中，微观管理者可能是对开发者的生产力最差的。当然，微观管理者往往有更多的会议和计划外的干扰。但这并不仅仅是这样。他们表现出缺乏信任，通过这样做，你会觉得他们不断地破坏你的技能和你完成事情的能力。开发人员在中断之间的任何动力在这时都会直接消失。这种影响超越了生产力。微观管理者可能是开发人员离开的第一个原因，或者至少是更换团队。

3) 含糊不清

有很多方法可以说明模糊性。像 "它坏了，修好它！"这样的错误报告没有足够的信息让开发人员据以工作。顺便说一下，在这种情况下，有一个错误报告模板可以帮助。

或者对某项功能的规范不明确，在这种情况下，开发人员会开始实施他们感觉正确的东西，一旦管理者更好地详细说明预期行为，他们就不得不从头开始。

不明确的优先次序也属于这个类别。开发人员花在怀疑他们是否正在从事正确的任务上的时间可以很容易地避免。如果有一天他们收到经理的评论，问他们为什么在这个特定的任务上工作（而优先级没有被定义）......好吧，你明白了--很多挫折......

4) 海鸥管理

你听说过这种情况吗？当经理们完全不参与工作时，就会发生这种情况，但是......他们只是偶尔俯冲下来，在所有东西上拉屎。"这是不对的，还有这个，这个看起来很糟糕，"等等，然后又飞走了。我不得不承认我很喜欢这个画面，但不幸的是，这种情况比我们希望的更经常发生。这种行为让开发者深感沮丧；他们在接下来的几个小时内将无法回到这个区域，有时甚至几天内都无法回到这个区域。

5) 贪婪的信贷

你是否曾有过一个经理或其他开发人员，把你在过去几周所做的工作的所有功劳都归于他？开发人员最看重的是能力。抢走别人的功劳，就是把别人的能力占为己有，从他或她那里拿走。这在我的名单上排得很靠前，因为我觉得这造成了很大的紧张，以至于在相当长的一段时间内破坏了整个开发者的生产力。

6) 环境 - 噪声、运动、工作空间设计...

这对非程序员来说可能很奇怪，但开发人员的工作环境对他们的活动有重要影响。例如，有一些白噪声--响亮的空调，听到汽车和卡车的滚动声--有助于他们更好地集中注意力。这就是为什么我们中的许多人都戴上了耳麦!实际上，我刚刚发现RainyMood--真的很棒👍!

同样地，如果工作空间被设计成有尽可能多的运动，这不会帮助他们集中注意力！或者将台式电脑屏幕的方向设计成对经理人高度可见。或者将台式电脑屏幕的方向设计成对管理人员高度可见......那么，这就会产生一些额外的压力，甚至更多被打断的机会。

7) 范围蠕变

项目管理中的范围蠕动（也称为焦点蠕动、需求蠕动、功能蠕动，有时也称为厨房水槽综合症）是指项目范围中不受控制的变化。当一个项目的范围没有被正确定义、记录或控制时，就会发生这种情况。

范围蠕变将相对简单的请求变成了可怕的复杂和耗时的怪物！而且大多数时候它发生在开发过程中。而且大多数时候，它发生在开发过程中!例如，对于一个简单的功能：第一版（实施前）：功能是 "显示一个地点的地图 "第二版（当第一版快完成时）：功能改为 "显示一个地点的3D地图 "第三版（当第二版快完成时）：功能再次改为 "显示一个用户可以飞过的地点的3D地图"

8) 产品定义过程

所以这个乍看之下可能很奇怪，但其实很容易理解。如果一个产品团队在定义其团队的优先级时，从来没有验证（通过客户反馈或任何其他方式）相应功能的利益，而开发人员看到大多数功能最终只是没有被使用，他们会觉得他们所做的事情是无用的，并会失去动力。我们都希望感到自己有影响力，而这对开发者来说可能更加重要!

9) 缺乏对技术债务的考虑

技术债务是指为了更快地发布软件而故意决定实施不是最好的解决方案或编写不是最好的代码。承担一些技术债务是不可避免的，在短期内可以提高软件开发的速度。然而，从长远来看，它有助于提高系统的复杂性，从而拖慢开发人员的速度。非程序员往往低估了生产力的损失，并倾向于总是向前推进，这就成了一个问题。但是，如果重构从来不是优先事项的一部分，它不仅会影响生产力，也会影响产品质量。

10) 工具的多重性和硬件

开发人员每天使用许多工具来编程、推送和合并他们的代码。自动化程度越高越好。不言而喻，如果你使用 "古老 "的工具，这将影响你的生产力。同样，拥有一个大屏幕与只有一台笔记本电脑也会产生影响。考虑到硬件的成本和开发人员的工资，只要有5%的生产力提高，在这一点上，任何投资都是值得的!只要给你的开发者团队喜欢的工具和硬件（单独的硬件，但作为一个团体的工具）。

11) "如何 "注释代码

当我们学习如何编码时，我们被告知要尽早和经常写注释。这个想法是，有太多的注释比有太少的注释要好。不幸的是，许多程序员错误地将此理解为他们必须对每一行代码进行注释，这就是为什么我们经常看到这样的代码（来自Jeff Atwood在"Coding Without Comments"上的文章）。

r = n / 2; // 设置r为n除以2

// 当r - (n/r)大于t时进行循环，而( abs( r - (n/r) )> t ) { r = 0.5 * ( r + (n/r)); // 设置r为r + (n/r)的一半

}

你知道这段代码是做什么的吗？我也不知道。问题是，虽然有很多注释描述了这段代码在做什么，但却没有描述它为什么要这样做。如果程序中有一个错误，而你偶然发现了这段代码，你将不知道从哪里开始。

12) 不可能实现的Deadline

这最后一个问题与经理们的倾向有关，他们要求开发人员进行估算，然后推动他们尽可能地降低这些估算，然后神奇地将其视为最后期限！经理们甚至会认为，由于开发人员自己 "决定"，他们承诺了最后期限。经理们甚至会认为，由于开发人员自己 "决定 "了估计值，他们承诺了最后期限，因此这些最后期限应该被认为是有效的，足以分享给上层管理人员。

毫不奇怪，开发人员觉得这些最后期限是不合理的，是任意的紧；这造成了紧张和无法集中注意力。

所有这些事情对开发人员来说是独一无二的吗？如果你看一下所有的12件事情，它们实际上对大多数其他基于项目的工作是很常见的。只是其中每一项的影响对开发者来说更加重要，因为他们需要深度专注才能在任务中取得进展。

如果你在你的公司内认识到上面提到的一些要点，与你的开发人员一起解决这些问题可能是有趣的。与他们交谈；找出这些是否是一个问题，以及如何解决它。无论他们说什么，最重要的是要相信他们的反馈和见解。虽然今天的技术与30年前有很大不同，但教训仍然是一样的。当你考虑团队生产力时，你不能忽视人的因素。与你的团队反复讨论你的流程、环境和工作习惯，让他们指导你如何拥有最高的生产力和影响力。
